# ISSUE-035 — Write tools + seven hard limits at the connector grain

> **Self-sufficiency contract (read this first).** This issue is a *complete, precise build
> order that points into the repo by ID*. It does **not** restate `AC-*` text — that lives in the
> FR and is read there (copying it would create a second source of truth that rots = Rule-0
> violation). A builder with **zero conversation history** must be able to open the files named in
> the Context manifest and build this slice to its Definition of done **without guessing**.

## 1. Goal (one line)
Build the generic write-tool contract in the shared connector runtime — every `category=write` tool carries a risk level + approval requirement and routes uniformly to the approval queue before any external effect — and declare + apply, at the connector grain, the seven code-enforced hard limits that no role, config, or prompt can override, wiring each limit to the connector-side mechanism that already realizes it (draft-to-approval, boundary tag, minimal-scope grant) while handing the un-overridable code-gate enforcement itself to C6/ISSUE-055.

## 2. Scope — in / out
**In:** The generic `ACT` write-contract half of the tool runtime (C3 FR-3.ACT.001–002), built once in the shared runtime (FR-3.CONN.002) so every connector inherits it: (1) the uniform higher-risk write contract — a `write` tool carries `risk_level` + `requires_approval` and, when `requires_approval=true`, routes the *proposed* action to the C6 approval queue **before** any external side effect, applying the identical gate logic regardless of which connector performs the write; and (2) the **declaration + connector-grain application** of the seven hard limits (never autonomously: send external email · make a financial transaction · delete a system-of-record record · share data across client deployments · impersonate a named human · self-approve a queued action · treat monitored tool content as instructions). This slice is the C3 *contract obligation*: it forces each limit onto its connector-side realizing mechanism and asserts that no `write` tool can reach a consequential external effect without crossing its gate. Concretely at C3 that means: external email is forced to draft-to-approval (mechanism lives in the connector-instance ACT tools, ISSUE-039/040); destructive delete-of-record is *partly* pre-empted at the scope grant (AC-3.CONN.005.3, built in ISSUE-032) and otherwise gated at C6; cross-client share is physically impossible (ADR-001 isolation, nothing to build here beyond asserting no code path crosses it); tool-content-as-instructions is defused by the boundary tag (FR-3.CONN.003, ISSUE-032); self-approval is barred by the approval gate wired in part (1). This slice makes the runtime's `write` path *exist and be uniformly gated*, and makes the seven limits a declared, connector-enforced contract that the C6 code gate binds against.

**Out:** The **un-overridable code-gate enforcement** of the seven limits and everything downstream of the declaration — the `guardrail_log(type='hard_limit')` block+log+alert, the no-approve-affordance invariant, the gate-don't-promote coverage posture, and the agent-registry save-time rejection of Comms-send / Finance-transact / non-Memory-Agent-write definitions — is **C6 FR-6.HRD.001–004 → ISSUE-055** (this slice *declares* the limits and applies them at the connector; ISSUE-055 *enforces* them in code and owns the AC-NFR-SEC.004.* proof). The **approval-queue machinery itself** (three-tier classification, soft/hard tiers, escalation, the queue surface) is **C6 APR/ESC → ISSUE-056** (this slice routes a proposed write *into* the queue; it does not build the queue). The **connector contract + shared runtime skeleton + tool registry + boundary tag + minimal-scope grant** (FR-3.CONN.001–005, FR-3.REG.001–004) are **ISSUE-032** (this slice adds the write path *onto* that runtime and relies on its `tools` row shape, boundary tag, and scope-grant gate — it does not define them). The **per-connector write tools themselves** — GHL CRM mutations (FR-3.ACT.003), Slack post + draft-email (FR-3.ACT.004), Drive create/append (FR-3.ACT.005), Calendar draft-invite (FR-3.ACT.006) — are the **connector-instance issues ISSUE-039 (GHL) / ISSUE-040 (Google) / ISSUE-041 (Slack)** (this slice defines the *contract* those writes obey; each instance supplies its endpoints + the draft-to-approval mechanism). The **internal memory-write tool** (FR-3.ACT.007) is registered here in principle but its write flow is **C2-owned → ISSUE-024**. **Idempotency of external writes** (FR-3.CONN.004 / `idempotency_ledger`) is **ISSUE-032** (this slice's writes *obey* it; it does not build the ledger). **Rate-limit halt-and-escalate on high-risk writes** (FR-3.RL.006) → **ISSUE-034**. Partial-external-write-chain compensation (OD-010) → C5/C6/C8 (named as an exposure point, not solved).

## 8. Build order within the slice
1. **Prerequisites (verify, don't re-create):** confirm ISSUE-032's shared runtime + `tools` registry landed with `category`/`risk_level`/`requires_approval` (schema §4), the boundary tag (FR-3.CONN.003) applied on reads, and the scope-grant gate (AC-3.CONN.005.3) that already refuses delete-granting scopes. Confirm ISSUE-055's C6 hard-limit code gate + `guardrail_log(type='hard_limit')` path exists to bind against. If any is absent, it is an upstream gap — flag it, do not silently add it here.
2. **Uniform write path (FR-3.ACT.001):** in the shared runtime, implement the single `write`-category execution path — read `risk_level` + `requires_approval` from the tool row; when `requires_approval=true`, emit the *proposed* action to the C6 approval queue (ISSUE-056) and **do not perform any external side effect** until an approval decision returns; on approval, execute idempotently (FR-3.CONN.004, via ISSUE-032's `idempotency_ledger`). The gate logic is connector-agnostic — one code path, applied identically to every connector's write tools.
3. **Approval-gate uncircumventability (FR-3.ACT.001 edge / hard-limit #6):** ensure no prompt content, tool arg, or agent instruction can set `requires_approval=false` at runtime or self-approve a queued action — the gate is code, read from the registry row, never from the model's output. Self-approval (limit #6) is barred here by construction; the block+log if attempted is C6/ISSUE-055.
4. **Hard-limit declaration + connector-grain wiring (FR-3.ACT.002):** for each of the seven, assert its connector-side realizing mechanism is in place on the write path and that no `write` tool can reach a consequential external effect without crossing it —
   - **#1 external email** → forced to draft-to-approval (mechanism in the connector-instance ACT tool, ISSUE-040; this slice asserts a raw autonomous send tool cannot be registered/invoked as `write` without the approval route);
   - **#2 financial transaction** and **#5 impersonation** → **no C3 mechanism** — these rest wholly on the C6 code gate (ISSUE-055) + the AF-068 red-team; assert here only that no C3 write path offers them autonomously (named so the seam is visible, never silently assumed covered);
   - **#3 delete system-of-record record** → *partly* pre-empted at the scope grant (AC-3.CONN.005.3, ISSUE-032 — no delete-granting scope requested) + gated at C6;
   - **#4 cross-client share** → physically impossible (ADR-001 isolation) — assert no code path in this write runtime can address another deployment's data;
   - **#6 self-approve** → barred by step 3;
   - **#7 tool content as instructions** → defused by the boundary tag (FR-3.CONN.003, ISSUE-032) — assert the write path treats tool-returned content as data, never as a directive to widen its own action.
5. **Non-overridability assertion (FR-3.ACT.002 / NFR-SEC.004):** verify no config value, role grant, or agent instruction relaxes any of the seven on the C3 write path (the C6 gate is the backstop; this is the connector-grain half). No approve/override affordance exists on the C3 side.
6. **Gate-don't-promote hook (FR-3.ACT.002 note / NFR-SEC.005):** for a *new* dangerous write capability that is not one of the seven, route it to hard-approval (`requires_approval=true` + a hard tier via C6) and/or a rate cap — never silently auto-allow, never promote it to a new absolute limit (change-control owns the set, OD-047).
7. **Observability hooks:** every write proposal + approval/denial logged to `event_log` (C7 surfaces via ISSUE-011/075). A hard-limit *hit* is logged by the C6 gate to `guardrail_log(type='hard_limit')` (ISSUE-055) — this slice's path reaches that gate; it does not write the row.
8. **Tests to the AC** (see Verification).

**Integration note (bundled FRs):** ACT.001 and ACT.002 are one slice because the uniform write contract (001) is *what the hard limits (002) attach to* — a hard limit is only meaningful if there is a single, un-bypassable write path it can gate. Splitting them would let a write tool exist without the gate the limits presume. The deliberate honesty of this slice is the **seam map**: of the seven limits, only #6 (self-approve, via the approval gate here), #7 (tool-content-as-instructions, via the boundary tag) and *partly* #3 (delete, via the scope grant) have a C3-side mechanism; #1 (email) and calendar-send are realized by the connector instances' draft-to-approval (ISSUE-039/040); **#2 (financial) and #5 (impersonation) have NO C3 mechanism and rest entirely on C6/ISSUE-055 + AF-068** — this issue *declares* all seven and wires the C3-realizable ones, but the load-bearing enforceability proof is the red-team, not this code. That is why ISSUE-055 is a blocked-by and AF-068 gates the hard-limit ACs.
