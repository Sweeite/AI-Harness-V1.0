# ISSUE-037 — Trigger infra + liveness (watch re-arm, event-gap reconciliation)

> **Self-sufficiency contract (read this first).** This issue is a *complete, precise build
> order that points into the repo by ID*. It does **not** restate `AC-*` text — that lives in the
> FR and is read there (copying it would create a second source of truth that rots = Rule-0
> violation). A builder with **zero conversation history** must be able to open the files named in
> the Context manifest and build this slice to its Definition of done **without guessing**.

## 1. Goal (one line)
Build the connector trigger layer — the once-per-connector inbound handler/parser, the no-code `event+condition→task` config, the per-vendor transport & verification wiring, and the liveness spine (proactive watch re-arm + event-gap detection/reconciliation) so a missed, dropped, or lapsed inbound event becomes a loud `degraded` condition, never silent knowledge loss.

## 2. Scope — in / out
**In:** The C3 TRIG slice — the generic, once-per-connector trigger pipeline that receives an **already-verified** webhook/event (from C0 webhook-auth, ISSUE-017) and normalizes it through the connector's parser to trigger evaluation (FR-3.TRIG.001); the end-user, no-code `event + condition → task` configuration and its runtime condition-matching (FR-3.TRIG.002); the per-connector default trigger set with per-deployment enable/disable toggles (FR-3.TRIG.003); the per-connector transport + inbound-verification *scheme wiring* — GHL native webhook Ed25519, Google Pub/Sub OIDC-JWT (Gmail) + signed channel-token (Drive/Calendar), Slack Events API HMAC — homing OD-044's per-vendor schemes into the connector contract (FR-3.TRIG.004); and the two liveness mechanisms this issue is named for: proactive watch/subscription re-arm before expiry with fail-loud on lapse (FR-3.TRIG.005) and event-delivery gap detection + watermark-based reconciliation (FR-3.TRIG.006). Detecting a **never-arriving** inbound webhook is homed here per OD-104(a) — it is a special case of the FR-3.TRIG.005/006 gap machinery, no new FR. On a matched trigger, this slice launches a task at the C5 boundary.

**Out:** Webhook *authentication* — raw-body/constant-time/`401`-reject signature verification — is C0's trust boundary (ISSUE-017, FR-0.WHK.*); this slice consumes the verified payload and applies the per-vendor scheme as the connector contract homes it (the algorithm identity, not the auth primitive). The connector contract/shared runtime and `tools` registry are ISSUE-032 (blocked-by). OAuth token lifecycle + the token-refresh job this re-arm job mirrors is ISSUE-033 (TOK). Rate-limit tiers are ISSUE-034 (RL). What a matched trigger's task *does* (queue, graph, execution) is C5 (ISSUE-047/048+). What re-ingested/reconciled events *become* is C2 ingestion (ISSUE-026, FR-2.ING.*). The concrete per-connector build wiring (GHL/Google/Slack read arms, real endpoints, dossier-specific params) lands in ISSUE-039/040/041, which consume this generic trigger infra. The trigger-config UI surface itself is a Phase-3 surface (see note); this slice owns the config data + runtime evaluation, not the screen. Connector `degraded`-state modelling, health panel, and alerts are ISSUE-038 (DSC) — this slice *raises* the `degraded` condition and *surfaces* watch/delivery health into that panel, but does not own the panel.

## 8. Build order within the slice
1. Trigger pipeline skeleton on the shared runtime (FR-3.TRIG.001): a once-per-connector inbound handler that accepts the **already-verified** event (C0 boundary), runs the connector's payload parser → normalized event, and hands it to trigger evaluation. Malformed/unparseable payloads are rejected and **logged, never silently dropped** (#3, AC-3.TRIG.001.2).
2. Per-vendor transport + verification-scheme wiring (FR-3.TRIG.004): route by connector and apply its scheme as homed by OD-044 — GHL `X-GHL-Signature` Ed25519 (dedup on `deliveryId`; legacy `X-WH-Signature` rejected post-2026-07-01; signing input per AF-090); Slack Events API HMAC-SHA256 over `v0:{ts}:{raw_body}` + >300s skew reject + `event_id` dedup; Gmail Pub/Sub OIDC-JWT (`aud`/`email`/skew); Drive/Calendar signed `X-Goog-Channel-Token` + TLS. Read the signing secret/public key/channel token from `connector_credentials` (FR-3.TOK.001, owned by ISSUE-033). **Hold the GHL arm until AF-090 GREEN.**
3. Default trigger set + per-deployment toggles (FR-3.TRIG.003): ship each connector's default triggers (GHL lead/stage/tag/overdue; Slack message/DM; Gmail new email; Calendar created/starting; Drive created/updated) as data in `tools.config`, each independently enable/disable-able; a disabled trigger fires nothing (AC-3.TRIG.003.2). Enabling a trigger whose scope/transport is missing surfaces the gap (FR-3.OPT.004 / FR-3.DSC.005 seam).
4. No-code config + runtime evaluation (FR-3.TRIG.002): persist user `event + condition → task` rules (Admin/authorized, default-deny; audited); on each normalized event evaluate conditions and launch the task when matched (seam to C5). Invalid/overlapping rules validated at save; missing-field conditions rejected.
5. Watch/subscription re-arm job (FR-3.TRIG.005): a scheduled job mirroring the token-refresh job (FR-3.TOK.002, ISSUE-033) that finds Google watches (Gmail Pub/Sub ~7d; Drive `files` 1d/`changes` 7d; Calendar bounded — no auto-renew) expiring within `CFG-watch_rearm_lead_minutes` and re-arms them, persisting the new channel/expiry. A **failed or missed** re-arm moves the connector to `degraded` (FR-3.DSC.001, ISSUE-038) and surfaces watch expiry on the health panel (FR-3.DSC.005) — never a silent stop. Slack/GHL non-expiring transports are skipped.
6. Event-gap detection + reconciliation (FR-3.TRIG.006): a scheduled sweep + delivery-health monitor. Slack — monitor own 2xx rate, flag approach to the 95%/60min auto-disable threshold as `degraded`, and periodically re-read `conversations.history` from the persisted per-channel `ts` watermark, re-ingesting gaps (feeds C2 ING). Google — a `history.list` 404 / `changes` token-expiry / `events.list` 410 triggers a full-sync from the last good watermark. A sweep that itself cannot run is alerted; the gap is never assumed empty. Sweep interval = `CFG-event_reconciliation_sweep_minutes`. **Slack arm held until AF-084/AF-083 GREEN.**
7. Never-arriving-webhook special case (OD-104(a)): confirm the FR-3.TRIG.005/006 machinery covers the missing-expected-trigger case (a lapsed watch or a delivery-gap watermark that never advances) — no new mechanism, just verify the `degraded`/alert path fires.
8. Test to the ACs: per-connector inbound normalize + malformed-reject (TRIG.001); scheme verification battery per vendor (TRIG.004); default-set toggle on/off (TRIG.003); rule match/no-match launches (TRIG.002); watch re-arm before expiry + fail-loud on missed re-arm + health-panel surfacing (TRIG.005); Slack gap sweep + Gmail 404 full-sync + delivery-threshold `degraded` flag (TRIG.006).

**Integration note (spans the bundled FRs):** the two halves of the trigger model are one pipeline — FR-3.TRIG.001 is the Layer-1 dev infra (handler/parser) and FR-3.TRIG.002/003 the Layer-2 dashboard config that rides on it; FR-3.TRIG.004 is the per-vendor transport/scheme that feeds the same handler. The liveness pair (FR-3.TRIG.005/006) is the load-bearing #3 guarantee for this whole component: without it an expired watch or a dropped/auto-disabled subscription looks identical to a genuinely quiet channel, and inbound knowledge silently stops. Build re-arm and gap-reconciliation as first-class jobs (not afterthoughts on the happy path) — both must resolve to a loud `degraded` condition (FR-3.DSC.001, ISSUE-038), and the never-arriving-webhook case (OD-104) is subsumed here rather than in C0, because auth ≠ liveness. Per-connector correctness (GHL Ed25519 base-string AF-090, Slack history affordability/recovery AF-083/084, Google watch/token expiry AF-108/109) is proven in the per-connector instances (ISSUE-039/040/041); this slice ships the generic mechanism with those arms gated.
